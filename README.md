# 事件质量验证系统 Demo

简化版事件验证系统 Demo。主要实现了事件的异步接收、多规则校验、模拟发送，以及多线程环境下的并发稳定性压测。

## 1. 整体架构设计

项目使用标准的 Maven 目录结构，主要分为以下几个模块：

* **数据模型 (`com.demo.model`)**
    * 定义了 `Event` 类。因为事件属性比较多，这里把属性设为不可变的，这样在多线程传来传去的时候比较安全。
* **校验规则 (`com.demo.rule`)**
    * 定义了一个通用的 `ValidationRule` 。然后把“必填校验”、“金额校验”、“货币正则校验”等分为了不同的实现类。这样做的好处是以后如果要加新规则，不用改以前的代码，直接加新类就行。
* **发送模块 (`com.demo.sender`)**
    * 定义了 `EventSender` 接口。写了两个实现：一个是简单的 `ConsoleSender`（控制台打印），另一个是 `MockNetworkSender`（在里面加了线程休眠模拟延迟，以及随机失败率）。
* **调度中心 (`com.demo.core`)**
    * `EventDispatcher` 类是整个流程的入口。为了满足题目“不阻塞主线程”的要求，我在里面建了一个**线程池**。主线程收到事件后直接扔给线程池去异步处理（校验 + 发送）。

## 2. 如何运行 Demo

本项目基于 **Java 11** 和 Maven 构建。

* **运行并发压测**：在 IDEA 中打开 `src/test/java/com/demo/ConcurrencyStressTest.java`，直接运行 main 方法。控制台会打印出 1000 个并发请求的处理过程和最终的统计结果。
* **运行单元测试**：打开 `src/test/java/com/demo/ValidatorTest.java`，运行测试类，即可验证具体的规则拦截逻辑（如负数金额、空字段等边界场景）。
* **命令行运行**：也可以在项目根目录下执行 `mvn clean compile` 和 `mvn clean test`。

## 3. 并发稳定性验证 (压测与对账)

为了验证系统在多线程下的稳定性，我在 `ConcurrencyStressTest` 里写了一个并发压测场景：

* **怎么压测**：用一个 for 循环结合 `new Thread` 瞬间起了 1000 个线程同时提交事件。为了模拟真实情况，我故意在里面混入了一些“脏数据”（比如金额是负数），并且启用了带 5% 失败率的 `MockNetworkSender`。最后用 `CountDownLatch` 等所有线程跑完。
* **怎么判定没有丢失**：我在处理类里用了 `LongAdder`（比普通的 `AtomicInteger` 性能更好）来统计各项数据。压测跑完后，代码会自动对比：`总接收数 == (发送成功数 + 发送失败数 + 校验被拦截数)`。如果两边相等，就证明在多线程下没丢数据。
* **怎么判定没有重复**：用了一个 `ConcurrentHashMap` 记录每个事件 ID 的处理次数。最后遍历一遍这个 Map，如果所有的次数都是 1，说明没有事件被重复处理，证明了线程安全。

## 4. 哪些地方可以继续改进

因为这只是一个本地的 Demo，如果放在真实的实际项目中，还可以从以下 3 点进行改进：

1. **压测方式比较简陋**：现在是用原生 Thread 写 for 循环来模拟并发，受限于电脑本身的性能。真实项目应该用 JMeter 等专业的压测工具来做大并发压测。
2. **缺少消息队列 (MQ) 缓冲**：目前事件是直接丢进本地内存的线程池里。如果并发量太大，可能会把内存撑爆或者触发拒绝策略。可以引入 Kafka 或 RabbitMQ，先把事件存起来慢慢消费。
3. **规则不能动态修改**：目前的校验规则（比如判断货币是3位大写字母）是写死在 Java 代码里的。以后可以把这些规则抽离到配置文件里，这样改规则就不用重新编译。

## 5. AI 工具使用说明

在完成这个作业的过程中，我主要使用了 AI 工具（Gemini）来辅助开发，提高编写代码的效率。在这个过程中，我的状态更像是一个“Reviewer”，具体的分工如下：

* **AI 主要帮我做了什么（负责写代码和查语法）**：
  1. **生成基础代码**：帮我快速写好了一些基础的 Java 类、接口定义和 Maven 配置。
  2. **查语法和 API**：向 AI 提问具体的语法细节，比如校验“3位大写字母”的正则表达式怎么写（`^[A-Z]{3}$`）。
  3. **提供优化建议**：一开始我打算用 `AtomicInteger` 来做并发计数，AI 建议我在高并发场景下使用 `LongAdder`，我查阅资料验证后采纳了。
  4. **排错**：遇到环境配置报错时，发给 AI 帮忙快速定位。

* **我的人工判断与核心工作（负责纠偏和把关）**：
  1. **纠正 AI 的架构思路**：最开始 AI 为了省事，直接给我生成了一份高度耦合的单文件代码。我觉得这样根本不符合题目中“易于扩展（可插拔）”的要求。所以我人工推翻了它的第一版，要求并引导 AI 按照策略模式，把代码拆分成了模型（Model）、规则（Rule）、发送（Sender）等不同模块，做定点改造。
  2. **严格的代码审查 (Code Review)**：AI 写的代码有时候会忽略题目的细节。我会仔细看它写的每一行核心逻辑是不是对的，比如：压测时的并发设计是不是真的线程安全（`ConcurrentHashMap` 的使用）、脏数据有没有被正确拦下来、最后的成功失败统计对不对得上面试题的验收标准。
